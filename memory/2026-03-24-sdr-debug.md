# SDR Bot Debugging - 2026-03-24 Evening

## The Real Problem

**The SDR agent can't respond because OpenClaw agents are LLMs, not script runners.**

When Telegram sends a message to the SDR bot:
1. ✅ Message arrives at Telegram binding
2. ✅ Binding routes to `agentId: "sdr"`
3. ✅ SDR agent is spawned
4. ❌ Agent has no instruction to call the skill/CLI
5. ❌ Agent generates a response using the LLM, which fails (no clear prompt)
6. ❌ User sees "typing" then silence

## Why Skills Didn't Work

- Created `/home/ubuntu/.openclaw/workspace-sdr/skills/sdr-automation/index.js` — but OpenClaw doesn't automatically call skills in Telegram messages. Skills are for agents to consciously decide to use them.
- Tried to make agent entry points (`/home/ubuntu/.openclaw/agents/sdr/agent/index.js`, `main.js`) — but OpenClaw doesn't execute these; agents are LLMs, not Node scripts.
- Tried to set SYSTEM_PROMPT files — but agents don't auto-load these either.

## Architecture Mismatch

OpenClaw agents are designed as **intelligent decision-makers** that:
- Receive user messages
- Think about what to do
- Decide which tools to use
- Generate responses

The SDR bot is supposed to be a **mechanical script** that:
- Always returns one of 6 hard-coded messages
- NO thinking
- NO decision-making
- Just keyword matching

These are fundamentally incompatible architectures.

##  Real Solutions (Require Code Changes Beyond This Session)

1. **Telegram Binding Hook**: Intercept SDR bot messages at the binding level before agent spawn, execute CLI, and respond directly without agent.
2. **Custom Agent Handler**: Create a custom OpenClaw agent handler that forces skill execution.
3. **Standalone Bot**: Don't use OpenClaw's Telegram integration; run the SDR bot separately as a Telegram bot with separate token.
4. **Webhook Integration**: Create a webhook endpoint that Telegram sends to directly, bypassing OpenClaw's agent system entirely.
