## SDR Automation Bot — LinkedIn URL Fix (ongoing session)

### Problem
Sales Navigator search result cards only expose `/sales/lead/ACwAAA...,NAME_SEARCH,xxx` URLs — never `/in/` URLs. The bot was sending Sales Nav links to Vaibhav (Telegram user 8176450202) instead of proper `linkedin.com/in/` profile URLs.

### Fixes attempted (in order)
1. **Swap querySelector priority** — swapped `/in/` before `/sales/lead/` in DOM selector. Didn't work — Sales Nav cards simply have no `/in/` links in their DOM.
2. **Response interceptor (recursive JSON)** — tried intercepting Sales Nav API responses and scanning for `entityUrn` + `publicIdentifier` at same JSON level. Didn't work — Sales Nav uses URN format `urn:li:fsd_salesProfile:(ACwAAA...,NAME_SEARCH,xxx)` which my regex didn't match, and `publicIdentifier` may be nested.
3. **Text-based proximity search** — improved to scan raw response text for entity IDs near `publicIdentifier` values. Still didn't work in practice.
4. **Browser navigation to lead page** (`resolveLeadUrl`) — navigate to each `/sales/lead/` URL and wait for API response or DOM link. Too slow (8s timeout per lead), and unreliable due to SPA rendering timing.
5. **LinkedIn Voyager API via `page.evaluate()`** — call `/voyager/api/identity/dash/profiles?q=memberIdentity&memberIdentity={entityId}` from within the open Sales Nav page. **Confirmed working via Python test** — returns correct `publicIdentifier` for all tested leads. However, the browser context navigates to `/sales/contract-chooser` instead of staying on the search page, causing `page.evaluate` execution context to be destroyed.

### Working API (confirmed)
```
GET /voyager/api/identity/dash/profiles?q=memberIdentity&memberIdentity={entityId}
Accept: application/vnd.linkedin.normalized+json+2.1
x-restli-protocol-version: 2.0.0
csrf-token: {JSESSIONID cookie value}
```
Returns `included[]` array — find item where `$type` includes `Profile` and has `publicIdentifier`.
