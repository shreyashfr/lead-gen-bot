## ARCHITECTURAL ROOT CAUSE IDENTIFIED — 18:11 UTC

**The SDR bot fails because OpenClaw agents are LLMs, not script executors.**

### What We Tried (All Failed)

1. **Skill-based approach** — Created `/home/ubuntu/.openclaw/workspace-sdr/skills/sdr-automation/index.js` with bare-minimum async function. OpenClaw doesn't automatically call skills for Telegram messages. Skills are for agents to consciously decide to use.

2. **Agent entry points** — Created `/home/ubuntu/.openclaw/agents/sdr/agent/index.js` and `main.js`. OpenClaw doesn't execute these; agents are spawned as LLM instances, not Node.js scripts.

3. **System prompts** — Created SYSTEM_PROMPT files. Agents don't auto-load these from workspace. No config field supports `systemPrompt` in agent definition.

4. **CLI tool approach** — Created `/home/ubuntu/.openclaw/workspace-sdr/bin/sdr-response` (works perfectly in isolation). But agent has no instruction to call it.

### The Real Flow (What Actually Happens)

```
Telegram message → Binding routes to SDR agent
    ↓
SDR agent is spawned as LLM (Claude Haiku)
    ↓
Agent receives message with ZERO context/instructions
    ↓
Agent has no tools, no clear task, no skill to reference
    ↓
Agent tries to generate a response
    ↓
Generation fails silently or times out
    ↓
User sees "typing..." then no reply
```

### Why "Typing Then Silence"

The agent IS being invoked (logs show it), but:
- It's an LLM with no instructions
- No system prompt is loaded
- No skill is automatically called
- Agent can't figure out what to do
- Response generation fails
- Telegram gets no message to send

### Why This Is An Architectural Mismatch

**SDR bot needs:**
- Mechanical execution (not thinking)
- Hard-coded responses (not generation)
- Keyword matching (not interpretation)
- State-based decisions (not chat context)

**OpenClaw agents provide:**
- Intelligent decision-making
- LLM-based generation
- Tool-mediated execution
- Context-aware responses

### Real Solutions (Require Gateway/Binding Code Changes)

1. **Telegram Binding Hook** — Intercept SDR bot messages at binding level, execute CLI directly, bypass agent spawn entirely.
2. **Custom Agent Handler** — Modify how OpenClaw spawns agents to detect SDR bot and force skill execution.
3. **Standalone Bot** — Run SDR bot outside OpenClaw with separate Telegram token.
4. **Webhook Endpoint** — Create HTTP endpoint that Telegram calls directly, no agent involved.
5. **ACP Harness** — Spawn SDR as a persistent ACP session with explicit "execute bin/sdr-response" instructions on every turn.
